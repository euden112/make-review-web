# Backend 파트 리팩터링 제안서

## 문서 목적
아키텍처 리뷰 결과를 바탕으로 Backend 파트에서 우선 반영이 필요한 개선사항을 정리합니다.
본 문서는 왜 수정이 필요한지(리스크)와 어떻게 수정해야 하는지(권장 조치)를 실행 가능한 형태로 제공합니다.

## 1) [공통 연계] 파일명/연동 인터페이스 계약 수립

### 리스크
- Crawler 산출물 규칙이 고정되지 않으면 ingestion 파이프라인에 예외 분기 로직이 누적되어 유지보수 비용이 증가합니다.

### 권장 조치 (Backend)
- ingestion 입력 계약(Contract)을 명시합니다.
  - 파일명: `{platform_code}_reviews_raw_{timestamp}.json`
  - 메타: `platform_code`, `collected_at`, `schema_version`, `record_count`
- `schema_version` 검증 실패 시 DLQ로 라우팅하고 알림을 발생시킵니다.

## 2) [협업 핵심] NLP 처리 책임을 Backend 비동기 워커로 이관

### 리스크
- NLP가 Crawler에 남아 있으면 수집 실패와 분류 실패가 결합되어 장애 경계가 불명확해집니다.

### 권장 조치 (Backend)
- Ingestion 비동기 워커에서 다국어/카테고리 분류를 수행합니다.
- 큐 기반 처리(Celery/RQ/내부 워커)로 재시도 및 장애 격리를 적용합니다.
- 실패 건은 DLQ에 저장하고 재처리 작업 경로를 분리합니다.

권장 처리 단계:
1. Raw 적재
2. 정규화/정제
3. NLP 분류
4. 실패 시 DLQ + 재시도
5. 성공 시 서빙 테이블 반영

## 3) [Backend/AI 협업] 스팸 전처리 로직 일원화

### 리스크
- Crawler `rule_based_filter`와 AI `is_spam_review`가 분리 유지되면 규칙 충돌 및 결과 불일치가 발생합니다.

### 권장 조치 (Backend)
- 공통 유틸리티로 규칙을 통합합니다.
  - 예: `cleaners/text_utils.py`
  - 함수: `normalize_text()`, `is_spam_review()`, `extract_language_hint()`
- Crawler/AI가 동일 규칙을 참조하도록 인터페이스를 고정합니다.
- 규칙 버전(`spam_rule_version`)을 이벤트/로그/저장 메타에 기록합니다.

## 4) [Backend 수용 전략] 필터링 개선안 반영 포인트

### 4-1. 원문 보존 원칙 수용
- Crawler가 원문을 전달하면 Backend는 원문 보존 컬럼과 노출용 요약 컬럼을 분리해 저장합니다.
- 길이 제한은 서빙 계층 또는 뷰 모델에서 적용합니다.

제거 영향 평가(Backend 관점):
- 데이터 품질: 장문 문맥 보존으로 분류/요약의 근거 품질이 향상될 가능성이 큽니다.
- 처리 성능: 비동기 워커의 전처리/분류 시간이 증가할 수 있어 워커 처리량(throughput) 저하 가능성이 있습니다.
- 토큰 비용: 원문 길이 증가로 map/reduce 입력 토큰이 늘어 AI 호출 비용이 증가할 수 있습니다.
- 저장 비용: 원문 보존에 따라 저장소 사용량과 I/O 비용이 증가할 수 있습니다.

운영 권장안(Backend):
1. 원문은 그대로 저장하고, AI 입력용 축약 텍스트를 별도 생성해 비용을 제어합니다.
2. 비정상 초장문 방어를 위해 안전 상한(예: 8000~12000자) 정책을 적용합니다.
3. AI 워커에서 문장 단위 발췌/요약 전처리를 적용합니다.
4. 변경 전후 지표를 비교해 정책을 점진 조정합니다.
  - 평균 본문 길이, 워커 처리량, AI 토큰 사용량, 실패율, 요약 품질 샘플 평가

### 4-2. UNIQUE_RATIO 완화 수용
- 장문 예외 기준(예: 400자)과 같은 규칙 파라미터를 중앙 설정으로 관리합니다.
- 정책 변경 시 코드 수정 없이 설정 배포로 반영 가능하게 합니다.

### 4-3. General 카테고리 수용
- `General`를 정식 카테고리로 허용합니다.
- 후속 재분류 배치가 해당 레코드를 우선 대상으로 삼도록 큐/잡 우선순위를 둡니다.

## 운영 지표(권장)
- ingestion_success_rate
- dlq_insert_rate
- nlp_worker_retry_count
- spam_reject_rate (by rule_version)
- general_category_ratio

## 완료 기준(DoD)
- 계약 위반 입력에 대한 명시적 실패 처리(DLQ/알림) 구현
- NLP 비동기 워커 운영(재시도/격리) 적용
- 스팸 규칙 공통 유틸리티화 및 버전 관리 적용
- General 카테고리 수용 및 후속 재분류 경로 확보
