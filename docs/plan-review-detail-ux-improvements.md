# 리뷰 상세 화면 개선안

> 작성일: 2026-05-21  
> 기준 화면: `demo.py` 실행 후 `/games/:id` 상세 페이지 스냅샷  
> 관련 문서: `docs/plan-purchase-desire-features.md`

## 1. 배경

현재 상세 화면은 프로젝트의 핵심 의도인 **구매 결정 지원**과 **취향 매칭**을 모두 포함하고 있다.

- 구매 결정 지원: 유저 리뷰 요약, 비평가 리뷰 요약, 장단점, 감성 분석, 카테고리 분석, 플레이타임별 여론
- 취향 매칭: 구매 타이밍, 추천 대상

문제는 두 목적의 정보가 화면에서 명확히 분리되지 않아, 사용자가 얻는 가치는 낮고 스크롤 비용은 커진다는 점이다. 특히 대표 리뷰와 감성 하이라이트가 비슷한 긴 원문 리뷰를 반복 노출하면서 기능 역할이 겹친다. 감성 하이라이트는 스포일러 위험도 있어, 현재 형태를 유지하기보다 더 안전한 취향 매칭 기능으로 대체하는 편이 적합하다.

## 2. 현재 문제점

### 2-1. 유저/비평가 요약의 정보량 부족

현재 유저 리뷰 요약과 비평가 리뷰 요약은 1~3문장 수준으로 표시된다. 이 정도 길이는 전체 여론의 방향은 알려주지만, 구매 판단에 필요한 구체적 근거를 충분히 제공하지 못한다.

문제:

- 유저들이 무엇을 좋게 봤는지 구체성이 부족하다.
- 불만이 어떤 사용자층에서 발생하는지 드러나지 않는다.
- 비평가 리뷰 요약이 유저 요약과 동등한 판단 축으로 보이기 어렵다.
- 점수와 감성 배지는 있지만, 그 점수를 해석할 수 있는 문맥이 부족하다.

관련 코드:

- `ai-pipeline/ai_module/map_reduce/reduce_api.py`: user summary가 2~4문장으로 제한됨
- `frontend/src/GameDetailPage.jsx`: 요약 본문을 단순 문단으로만 표시

### 2-2. 대표 리뷰와 감성 하이라이트의 역할 중복 및 폐지 필요성

문서상 대표 리뷰와 감성 하이라이트의 목적은 다르다.

| 영역 | 의도 |
|---|---|
| 대표 리뷰 | 신뢰 근거, 판단 근거 |
| 감성 하이라이트 | 감정 피크, 구매 욕구 유발 |

현재 화면에서는 둘 다 긴 긍정 원문 리뷰를 보여주는 방식이어서 사용자는 같은 정보를 반복해서 보는 것처럼 느낀다.

검토 결과, 감성 하이라이트는 서비스의 필수 기능으로 보기 어렵다. 구매 결정 지원에는 직접 기여가 약하고, 취향 매칭 목적은 원문 리뷰 캐러셀보다 안전한 추상화 콘텐츠로 대체할 수 있다.

문제:

- 대표 리뷰가 “왜 대표인지” 설명하지 않는다.
- 감성 하이라이트가 대표 리뷰와 다른 고유 가치를 충분히 제공하지 않는다.
- 동일하거나 유사한 리뷰가 두 섹션에 걸쳐 반복될 가능성이 있다.
- 대표 리뷰가 너무 길면 이후 분석 섹션 접근성이 크게 떨어진다.
- 감성 하이라이트는 스포일러 리스크를 감수하면서까지 유지할 만큼 핵심 가치가 크지 않다.

관련 코드:

- `backend/app/services/ai_service.py`: 대표 리뷰는 플랫폼별 score 기반으로만 선별
- `backend/app/api/v1/highlights.py`: 감정 키워드, helpful_count 기반으로 감성 하이라이트 선별
- `frontend/src/GameDetailPage.jsx`: 두 섹션 모두 원문 중심 카드 UI

### 2-3. 감성 하이라이트의 스포일러 위험

감성 하이라이트는 플레이어가 강하게 반응한 리뷰를 노출한다. 이때 리뷰가 엔딩, 특정 보스, 반전, 후반 지역, 캐릭터 사망, 퀘스트 결말을 언급하면 구매 욕구 유발 기능이 오히려 게임 경험을 훼손할 수 있다.

문제:

- "명장면"이라는 섹션명 자체가 구체적 장면 공개를 암시한다.
- 감정 강도가 높은 리뷰일수록 엔딩·반전·후반부 경험을 언급할 가능성이 높다.
- 현재 highlights API는 스포일러 여부를 판정하지 않는다.
- 원문 리뷰를 그대로 노출하면 프론트에서 사후 보호하기 어렵다.

결론:

- 감성 하이라이트는 기본 상세 화면에서 제거한다.
- 기존 `highlights` 로직은 당장 삭제하지 않더라도, 사용자 노출 기능이 아닌 내부 참고 또는 향후 실험 기능으로 격하한다.
- 취향 매칭 목적은 `이런 사람에게 추천` 섹션으로 대체한다.

관련 코드:

- `backend/app/api/v1/highlights.py`: 감정 키워드 기반 점수화만 수행
- `frontend/src/GameDetailPage.jsx`: 하이라이트 텍스트를 기본 노출

### 2-4. 긴 대표 리뷰 접기/펼치기 부재

대표 리뷰 카드가 긴 원문을 그대로 렌더링한다. 스냅샷 기준으로 Steam 대표 리뷰 1개가 화면 대부분을 차지한다.

문제:

- 상세 페이지의 정보 구조가 무너진다.
- 사용자는 요약, 장단점, 구매 타이밍까지 도달하기 전에 이탈할 수 있다.
- 긴 리뷰를 읽고 싶은 사용자와 빠르게 판단하고 싶은 사용자의 요구를 동시에 만족하지 못한다.

관련 코드:

- `frontend/src/GameDetailPage.jsx`의 `ReviewCard`
- 감성 하이라이트에는 `line-clamp-5`가 적용되어 있으나 대표 리뷰에는 유사한 제한이 없음

### 2-5. 대표 리뷰 선정 대상의 정확성 리스크

현재 대표 리뷰는 AI 파이프라인 실행 시 `new_reviews`를 기준으로 선별된다. 증분 요약 상황에서는 전체 리뷰 중 대표가 아니라 새로 들어온 리뷰 중 대표가 될 수 있다.

문제:

- 오래된 고품질 리뷰가 대표 리뷰에서 사라질 수 있다.
- “대표 리뷰”라는 라벨의 의미와 실제 데이터 기준이 어긋난다.
- 재실행 시점에 따라 대표 리뷰가 불안정하게 바뀔 수 있다.

관련 코드:

- `backend/app/services/ai_service.py`
- `_select_platform_representative_reviews(new_reviews, ...)`

### 2-6. AI 종합 요약 생성/폐기 정책 불일치

Reduce 프롬프트는 `full_text`를 생성하도록 요구하지만, 저장 단계에서는 `summary_text=None`으로 폐기한다.

문제:

- 모델 출력 토큰을 사용하지만 화면에서는 활용하지 않는다.
- “AI 종합 요약”을 제거한 현재 UI 방향과 프롬프트 요구사항이 맞지 않는다.
- 향후 유지보수 시 `full_text`가 실제 기능인지 폐기된 필드인지 혼동될 수 있다.

관련 코드:

- `ai-pipeline/ai_module/map_reduce/reduce_api.py`: `full_text` 생성 요구
- `backend/app/services/ai_service.py`: `summary_text=None`

### 2-7. 빈 데이터 상태가 화면 면적을 과도하게 차지

Metacritic 대표 리뷰나 비평가 요약이 없을 때 빈 카드가 크게 남는다.

문제:

- 실제로 제공되는 정보보다 UI가 더 무겁게 보인다.
- “데이터 없음” 상태가 반복되면 완성도가 낮아 보인다.
- 데이터가 없는 플랫폼과 데이터가 있는 플랫폼의 우선순위가 동일하게 표시된다.

### 2-8. 분석 섹션 간 정보 반복

장점/단점, 감성 분석, 카테고리별 분석, 플레이타임별 여론이 모두 긍정/부정 판단을 다룬다. 각각의 목적은 다르지만 현재 UI에서는 계층이 약하다.

문제:

- 사용자가 어떤 섹션을 먼저 봐야 하는지 불명확하다.
- 핵심 판단 정보가 여러 카드에 분산된다.
- 화면 스크롤 길이에 비해 새 정보량이 낮아질 수 있다.

## 3. 개선 원칙

### 3-1. 판단 근거와 욕구 유발을 분리한다

대표 리뷰는 구매 판단을 돕는 근거로 유지한다. 감성 하이라이트는 기본 화면에서 제거하고, 취향 매칭은 스포일러 없는 추상화 콘텐츠인 `이런 사람에게 추천`으로 대체한다.

### 3-2. 요약은 더 풍부하게, 원문은 더 짧게

현재 화면의 가장 큰 문제는 요약이 짧고 원문이 길다는 점이다. 방향을 반대로 바꾼다.

- 요약: 구체적 근거를 포함해 확장
- 원문: 기본 접힘, 필요 시 펼침

### 3-3. 빈 데이터는 크게 보여주지 않는다

없는 정보를 큰 카드로 노출하지 않는다. 작은 상태 표시나 섹션 생략으로 처리한다.

### 3-4. 한 화면의 각 섹션은 고유한 질문에 답해야 한다

각 섹션의 역할을 다음처럼 고정한다.

| 섹션 | 답해야 하는 질문 |
|---|---|
| 유저 리뷰 요약 | 실제 플레이어들은 무엇을 좋고 나쁘게 봤나? |
| 비평가 리뷰 요약 | 전문가들은 어떤 기준으로 평가했나? |
| 대표 리뷰 | 이 판단을 뒷받침하는 실제 리뷰 근거는 무엇인가? |
| 장단점/카테고리 | 구매 판단에 필요한 항목별 체크리스트는 무엇인가? |
| 플레이타임별 여론 | 초반/중반/후반 경험이 어떻게 달라지는가? |
| 구매 타이밍 | 지금 사도 되는가? |
| 이런 사람에게 추천 | 어떤 취향의 사용자에게 이 게임이 잘 맞는가? |

## 4. 제안 변경안

### 4-1. 유저/비평가 요약을 구조화된 카드로 확장

현재 단락형 요약을 다음 구조로 바꾼다.

유저 리뷰 요약:

- 한줄 결론
- 좋아한 점 3개
- 아쉬운 점 2개
- 추천 대상
- 주의할 사용자

비평가 리뷰 요약:

- 평론 핵심 관점
- 호평 포인트 3개
- 비판 포인트 1~2개
- 유저 평가와 별도로 참고할 점

API 응답도 단순 `summary` 문자열 대신 구조화된 필드를 추가하는 방향이 좋다.

예시:

```json
{
  "verdict": "방대한 오픈월드와 전투 설계가 강점이지만 난이도와 초반 진입 장벽은 분명합니다.",
  "positive_points": ["탐험 보상", "보스전 성취감", "빌드 다양성"],
  "negative_points": ["불친절한 안내", "일부 최적화 문제"],
  "recommended_for": ["도전적인 액션 RPG를 선호하는 유저"],
  "caution_for": ["명확한 퀘스트 안내를 기대하는 유저"]
}
```

단기 구현은 기존 `summary` 문자열을 5~7문장으로 늘리는 방식으로 시작할 수 있다. 장기 구현은 구조화 응답으로 전환한다.

### 4-2. 대표 리뷰를 “근거 리뷰”로 재설계

대표 리뷰는 긴 원문 카드가 아니라 판단 근거 카드로 바꾼다.

변경 전:

- Steam 대표 리뷰 3개
- Metacritic 대표 리뷰 3개
- 원문 중심

변경 후:

- 쟁점별 대표 리뷰
- 짧은 인용 1~2문장
- `왜 이 리뷰가 대표인지` reason 표시
- 전문은 펼침 또는 모달에서 확인

예시 카드:

```text
전투 시스템
"보스전은 반복 실패 끝에 패턴을 익히는 성취감이 강하다."
근거: 전투/난이도 관련 긍정 리뷰 중 공감 수가 높음
```

필요한 변경:

- 대표 리뷰 선별 시 `review_categories_json` 활용
- 중복 카테고리보다 다양한 쟁점을 우선
- 원문 전체가 아니라 `quote_excerpt`를 별도 생성하거나 프론트에서 절단

### 4-3. 감성 하이라이트 폐지 및 “이런 사람에게 추천”로 대체

감성 하이라이트는 기본 상세 화면에서 제거한다. 대신 리뷰 원문을 직접 보여주지 않고, 여러 리뷰에서 반복되는 매력 요소를 스포일러 없이 추상화한 `이런 사람에게 추천` 섹션을 추가한다.

변경 전:

- 원문 리뷰 캐러셀
- 감정 키워드와 helpful_count 기반 선별
- 스포일러 가능성 있는 문장 기본 노출
- 대표 리뷰와 정보 중복

변경 후:

- 스포일러 없는 추천 대상 3~5개
- 원문 직접 노출 없음
- 각 포인트는 카테고리, 리뷰 근거 수, 짧은 설명으로 구성
- 필요 시 대표 리뷰 섹션의 근거 리뷰와 연결

API 필드 제안:

```json
{
  "recommendations": [
    {
      "label": "탐험 보상",
      "category": "content",
      "summary": "숨겨진 지역과 아이템을 발견하는 재미가 반복적으로 언급됩니다.",
      "evidence_count": 48,
      "sentiment": "positive"
    },
    {
      "label": "전투 성취감",
      "category": "controls",
      "summary": "어려운 전투를 반복 학습으로 돌파하는 만족감이 강점으로 나타납니다.",
      "evidence_count": 36,
      "sentiment": "positive"
    }
  ]
}
```

선별 기준:

- `review_categories_json`의 긍정 카테고리 빈도
- helpful_count가 높은 긍정 리뷰의 반복 표현
- 유저 요약의 positive_points 또는 pros와의 정합성
- 스포일러 키워드가 포함된 원문은 근거로 사용하더라도 요약문에는 구체 사건을 포함하지 않음

기존 `GET /api/v1/games/{game_id}/highlights`의 처리:

- 단기: 프론트에서 호출하지 않음
- 중기: `/recommendation-targets` 또는 `/recommendation-targets` API로 대체
- 장기: 사용되지 않으면 제거

### 4-3b. 추천 대상의 스포일러 보호

대체 섹션은 원문을 노출하지 않지만, AI 요약문이 스포일러를 생성할 가능성은 여전히 있다. 따라서 출력 규칙을 명시한다.

규칙:

- 특정 보스명, 엔딩, 반전, 캐릭터 사망, 후반 지역명, 퀘스트 결말을 언급하지 않는다.
- 구체 사건 대신 경험 유형으로 표현한다.
- 리뷰 원문을 직접 인용하지 않는다.
- 스포일러 키워드가 있는 원문은 positive signal 집계에는 사용할 수 있지만 출력 문장에는 포함하지 않는다.

스포일러 키워드 예시:

```text
엔딩, 최종 보스, 마지막 보스, 반전, 죽음, 사망, 배신, 정체, 후반부,
ending, final boss, last boss, plot twist, dies, death, betrayal, true identity
```

표현 예시:

| 나쁜 표현 | 좋은 표현 |
|---|---|
| "최종 보스 OOO를 쓰러뜨리는 장면이 압권입니다." | "후반부 전투의 성취감이 강하게 언급됩니다." |
| "캐릭터 OOO의 죽음이 감동적입니다." | "서사적 여운과 캐릭터 몰입감이 장점으로 나타납니다." |
| "엔딩 반전이 놀랍습니다." | "마지막까지 긴장감을 유지한다는 반응이 있습니다." |

중요한 점은 추천 대상을 설명하되, 사용자가 아직 경험하지 않은 구체 사건을 알려주지 않는 것이다.

### 4-4. 긴 리뷰 접기/펼치기 추가

대표 리뷰 카드 기본 상태:

- 6~8줄까지만 표시
- `전체 보기` 버튼
- 펼친 상태에서 `접기` 버튼
- 원문/번역 토글은 유지

긴 원문 전문은 카드 안에서 무한히 늘리는 것보다 모달 또는 드로어로 여는 방식을 권장한다.

단기 구현:

- `ReviewCard`에 `expanded` state 추가
- CSS `line-clamp` 적용
- 일정 글자 수 이상일 때만 버튼 표시

### 4-5. 대표 리뷰 선정 기준 수정

현재:

```python
representative_reviews = _select_platform_representative_reviews(
    new_reviews,
    steam_pid,
    meta_pid,
    limit_per_platform=3,
)
```

권장:

```python
representative_reviews = _select_platform_representative_reviews(
    summary_reviews,
    steam_pid,
    meta_pid,
    limit_per_platform=3,
)
```

감성 하이라이트를 제거하는 방향이므로, 대표 리뷰와 `highlights` 간 중복 제거는 별도 구현하지 않는다. 대신 `highlights` API를 상세 화면에서 호출하지 않도록 정리한다.

### 4-6. `full_text` 정책 정리

둘 중 하나를 선택해야 한다.

안 A: `full_text` 폐지

- 현재 UI 방향과 맞음
- reduce 출력 토큰 절감
- 요약 구조가 user/critic 중심으로 단순해짐

안 B: `full_text` 복원

- “AI 종합 판단” 섹션으로 활용
- 유저/비평가 요약 위에 3~4문장 노출
- 단, 유저 요약과 중복되지 않도록 프롬프트 재작성 필요

현재 제품 방향에는 **안 A**가 더 적합하다. 이미 UI 주석에도 “AI 종합 요약은 유저 리뷰 요약과 중복이라 제거”라고 되어 있으므로, 프롬프트와 데이터 모델도 그 방향에 맞추는 편이 일관적이다.

### 4-7. 빈 데이터 상태 축소

Metacritic 데이터가 없을 때:

- 2열 빈 카드 유지 대신 Steam 단일 컬럼으로 확장
- 비평가 요약 없음은 작은 상태 배지로 표시
- 플랫폼별 대표 리뷰 섹션 제목에 실제 제공 플랫폼 수 표시

예시:

```text
플랫폼별 근거 리뷰
Steam 리뷰 기준 · Metacritic 데이터 없음
```

### 4-8. 분석 섹션 재배치

권장 순서:

1. 키워드
2. 유저 리뷰 요약 / 비평가 리뷰 요약
3. 구매 판단 체크리스트: 장점, 단점, 카테고리 분석 통합
4. 근거 리뷰
5. 플레이타임별 여론
6. 구매 타이밍
7. 이런 사람에게 추천

이 순서는 구매 결정 지원을 먼저 끝내고, 취향 매칭 콘텐츠를 하단에서 강화하는 구조다.

## 8. Reduce 응답 사용 현황 점검

현재 reduce 단계는 실제 화면에서 사용하지 않거나, 개선 방향과 맞지 않는 응답을 일부 생성하고 있다.

### 8-1. 명확히 불필요한 응답

| reduce 출력 | 현재 처리 | 판단 |
|---|---|---|
| `unified.full_text` | 생성은 필수로 요구하지만 `GameReviewSummary.summary_text=None`으로 저장하지 않음. 선택적으로 semantic similarity 계산에만 사용 | 폐지 권장 |

근거:

- `ai-pipeline/ai_module/map_reduce/reduce_api.py`는 `full_text`를 필수 출력으로 요구한다.
- `backend/app/services/ai_service.py`는 unified 본문 폐지 정책에 따라 `summary_text=None`으로 저장한다.
- 프론트는 `summary.one_liner`를 히어로에 사용하고, `summary.summary_text`는 fallback으로만 참조한다.

정리 방향:

- reduce 프롬프트와 validation에서 `full_text` 요구를 제거한다.
- `FinalSummary.full_text`를 optional 또는 제거 대상으로 바꾼다.
- semantic similarity 평가가 필요하면 `one_liner + pros + cons + user.summary + critic.summary` 기반으로 대체한다.

### 8-2. 생성·저장·응답되지만 현재 UI에서 거의 쓰지 않는 응답

| reduce 출력 | 저장/API | 현재 프론트 사용 | 판단 |
|---|---|---|---|
| `user.pros` | `UserSummary.pros`, `/user-summary` 응답 | 상세 화면 미사용 | 구조화 요약 개선 시 사용하거나 제거 |
| `user.cons` | `UserSummary.cons`, `/user-summary` 응답 | 상세 화면 미사용 | 구조화 요약 개선 시 사용하거나 제거 |
| `user.keywords` | `UserSummary.keywords`, `/user-summary` 응답 | 상세 화면 미사용 | 우선순위 낮음 |
| `critic.pros` | `CriticSummary.pros`, `/critic-summary` 응답 | 상세/비교 화면 미사용 | 구조화 요약 개선 시 사용하거나 제거 |
| `critic.cons` | `CriticSummary.cons`, `/critic-summary` 응답 | 상세/비교 화면 미사용 | 구조화 요약 개선 시 사용하거나 제거 |
| `critic.keywords` | `CriticSummary.keywords`, `/critic-summary` 응답 | 상세/비교 화면 미사용 | 우선순위 낮음 |
| `playtime.*.keywords` | `PlaytimeAnalysis.*_keywords`, `/playtime-analysis` 응답 | 상세/비교 화면 미사용 | 제거 후보 |

주의:

- `playtime.*.pros`와 `playtime.*.cons`는 상세 화면의 플레이타임 카드에서 사용한다.
- `unified.pros`, `unified.cons`, `unified.keywords`, `unified.aspect_scores`, `unified.sentiment_score`는 현재 상세/비교 화면에서 사용한다.
- `critic.summary`와 `user.summary`는 현재 상세 화면에서 사용한다.

### 8-3. 개선 문서 기준으로 새로 필요한 응답

`이런 사람에게 추천` 섹션은 현재 reduce 출력에 없다. 새 기능을 구현하려면 다음 중 하나를 선택해야 한다.

안 A: reduce에서 `recommendations`를 생성

```json
{
  "recommendations": [
    {
      "label": "탐험 보상",
      "category": "content",
      "summary": "숨겨진 지역과 아이템을 발견하는 재미가 반복적으로 언급됩니다.",
      "evidence_count": 48
    }
  ]
}
```

장점: 기존 AI 요약 흐름에 자연스럽게 통합된다.  
단점: reduce 응답 스키마가 커진다.

안 B: 비-LLM 집계로 생성

- `review_categories_json`
- helpful_count
- 긍정 리뷰 빈도
- 기존 `summary.pros`

위 데이터를 조합해 API에서 생성한다.

장점: 비용 증가가 없다. 스포일러 위험이 낮다.  
단점: 문장 품질은 reduce 생성보다 제한된다.

현재 방향에는 **안 B를 먼저 적용**하는 것이 적합하다. `이런 사람에게 추천`는 구체 사건을 말하면 안 되므로, LLM 자유 생성을 줄이고 카테고리 기반 템플릿으로 시작하는 편이 안전하다.

### 8-4. 정리 우선순위

| 우선순위 | 작업 |
|---|---|
| P0 | `full_text` 필수 생성 제거 |
| P0 | 상세 화면에서 `highlights` 호출/렌더 제거 |
| P1 | `user/critic.pros/cons`를 UI에서 실제 사용하도록 구조화 카드 개선 |
| P1 | 사용하지 않을 `user/critic.keywords`, `playtime.keywords` 제거 여부 결정 |
| P1 | `이런 사람에게 추천`를 카테고리 기반 API로 추가 |

## 9. 결론

현재 화면의 문제는 정보가 절대적으로 부족한 것보다, 정보의 밀도와 역할 분리가 약한 데서 온다. 요약은 더 구체적으로 만들고, 원문은 접어서 필요한 사람만 읽게 해야 한다.

우선 구현은 다음 세 가지로 압축한다.

1. 대표 리뷰 접기/펼치기
2. 유저/비평가 요약 확장
3. 감성 하이라이트 제거 및 `이런 사람에게 추천`로 대체

이 세 가지가 해결되면 상세 페이지는 프로젝트 의도인 구매 결정 지원과 취향 매칭을 더 명확하게 수행할 수 있다.
