# 패치노트 — 근거 기반 요약 + 클라우드 배포 통합

이 문서는 `feat/evidence-grounded-scoring`(요약 품질·근거화 작업)과 `feature/cloud-deploy`(클라우드 배포·운영 작업)를 `integration/feat-cloud` 브랜치로 합친 통합 릴리스의 변경 사항과, 이후 `main`에 반영된 요약 품질 개선을 정리합니다. 병합 기준점은 `df79e06`, 머지 커밋은 `09b5c06`입니다.

---

## 개요

이번 통합은 리뷰 요약의 **품질·근거화** 작업과 클라우드 **배포·운영**(인증, 키 로테이션) 작업을 하나로 합치는 것이 목표였습니다. 두 작업은 다루는 영역이 달라 대부분 충돌 없이 병합되었고, 두 쪽이 모두 Groq 클라이언트를 다루는 일부 코드에서만 조정이 필요했습니다.

통합 이후에는 항목 분류 품질을 더 다듬어, 콘텐츠 볼륨(`content`)과 스토리·캐릭터(`story`)를 분리하고, aspect 점수에 baseline과 근거 수를 함께 저장하도록 변경했습니다. 프론트엔드는 이 정보를 사용해 공통 레이더 축과 "눈에 띄는 반응" 보조 항목을 구분해서 표시합니다.

## ✨ 추가된 기능 (Added)

- **"이런 사람에게 추천"(recommended_for)**: 리뷰 근거를 바탕으로 게임별 추천 플레이어 유형과 그 사유를 생성해 저장하고 화면에 노출합니다. 이를 위해 `recommended_for_json`과 `caution_for_json` 컬럼을 추가했습니다(마이그레이션 14).
- **Steam 인기 태그(장르)**: 상점 페이지의 인기 태그를 수집해 장르 칩과 장르 필터에 사용합니다(`platform_meta_json.tags`). LLM이 추출하는 키워드와는 별개로, 일관된 장르 신호를 제공합니다.
- **Map 라우팅 정책 (`--map-route auto`)**: 토큰이 많이 드는 첫·전체 재처리는 로컬 Map으로, 양이 적은 증분은 Groq Map으로 보냅니다.
- **Map cache와 Reduce payload artifact**: 로컬 `run_map_pipeline.py`도 Redis chunk cache를 사용하며, 첫 요약·`--force` 전체 재처리에서는 `/reduce` 요청 payload를 `ai-pipeline/artifacts/reduce_payloads/keep/`에 보존합니다. 저장된 payload는 `--from-payload`로 Map 없이 Reduce만 재실행할 수 있습니다.
- **API 키 인증**: 비용이 발생하는 쓰기 엔드포인트(`reviews-for-map`, `reduce`, `summarize`, 리뷰 적재)에 `X-API-Key`를 요구합니다.
- **Groq 키 로테이션**: 429(요청 초과)를 감지하면 다음 키로 자동 전환합니다(`GROQ_API_KEYS`에 여러 키 등록 가능).
- **스토리/캐릭터 aspect 분리**: 기존에는 스토리·세계관·캐릭터 반응이 `content`에 섞였지만, 이제 `story` aspect로 별도 추출·저장·노출합니다. `content`는 콘텐츠 양, 퀘스트, 엔드게임, 이벤트, 파밍, 반복 플레이처럼 할 거리와 볼륨 평가에 집중합니다.
- **aspect 보조 메타데이터**: `aspect_sentiment_json`에 `baseline_score`, `evidence_count`, `evidence_review_ids`를 보존합니다. 프론트엔드는 이 값으로 보조 항목 노출 여부를 판단합니다.

## 🔧 변경된 기능 (Changed)

- **로컬 Map 모델**을 `qwen2.5:7b`에서 `gemma4:e4b`로 교체했습니다. A/B 테스트에서 약 18% 더 빠르고 JSON 형식 준수도 더 안정적이었습니다.
- **항목별(aspect) 점수 산출** 방식을 바꿔, 긍정/부정 판정을 Map LLM이 뽑은 근거의 polarity를 우선 사용하도록 했습니다(크롤러 카테고리 누적치는 보완용으로만 사용). 이전에는 크롤러 태그의 긍정 편향 때문에 불만이 많은 항목이 오히려 높은 점수를 받아 순위가 뒤집히는 문제가 있었는데, 이를 교정했습니다.
- **Reduce 개선 재실행 경로**를 추가했습니다. 일반 스케줄러 증분은 JSON artifact를 남기지 않고 Redis cache만 사용하지만, 커서가 없는 첫 요약이나 수동 `force` 실행은 payload를 영구 보존해 Reduce 프롬프트·로직 변경 시 오래 걸리는 Map을 다시 돌리지 않습니다.
- **카테고리 레이더**를 절대 점수가 아닌 게임 내 상대 능력치로 표현하도록 바꿨습니다. min-max로 펴되 과장을 막기 위한 하한을 두고, 축마다 강점·보통·약점 등급을 표시하며, 데이터가 부족한 축은 중앙으로 함몰시킵니다.
- **Steam 리뷰 수집**을 최신순 단독에서 최신순과 도움순(전체 기간)을 함께 모으는 방식으로 바꿨습니다(도움순 우세). 두 풀을 합칠 때 중복을 제거합니다.
- 화면의 **"키워드" 항목을 "리뷰 토픽"으로 이름을 바꿔** 장르 칩과 역할을 분리했습니다.
- **보조 지표 UI를 "눈에 띄는 반응"으로 변경**했습니다. `story`, `difficulty`, `sound`, `price_value` 중 근거 수가 충분하고 baseline 대비 차이가 큰 항목만 표시하며, 숫자 점수와 근거 수 대신 "좋게 언급되는 편" 또는 "아쉽게 언급되는 편"만 노출합니다.
- **카테고리 레이더 보정 폭을 완화**했습니다. 공통 5축(`content`, `gameplay`, `graphics`, `controls`, `optimization`)은 고정하되 하한과 보통 판정 기준을 높여 작은 점수 차이가 과장되어 보이는 문제를 줄였습니다.
- **게임 목록**을 발표용 100개 타이틀로 교체했습니다.
- **이미지 URL** 취득을 IStoreBrowseService API 기반으로 전환했습니다.

## 🐛 수정된 문제 (Fixed)

- **플레이타임 마지막 구간이 무너지는 문제**: 초반·중반·후반을 한 번에 생성할 때 근거가 충분한 마지막 구간(주로 후반)이 빈 장단점과 형식적인 문장으로 채워지곤 했습니다. 이제 장단점이 모두 비면 그 구간만 다시 호출하고, 그래도 비면 null(데이터 부족)로 처리합니다.
- **근거가 부족한 구간에 형식적 문장이 노출되는 문제**: 근거가 5개 미만인 구간에 LLM이 채운 형식적 문장을 저장하던 동작을 바꿔, "데이터 부족"으로 비워 둡니다.
- **추천 문구에 근거 ID가 노출되는 문제**: `(review_id=N)` 표기가 사용자에게 보이던 것을 저장 시 제거합니다.
- **모든 게임의 추천이 똑같던 문제**: 카테고리별 고정 문구 대신 게임별 실제 데이터로 교체했습니다.
- **스토리 평가가 콘텐츠 볼륨 점수를 흔드는 문제**: 스토리·캐릭터·세계관 평가를 `story`로 분리하고, 기존 크롤러 백필 매핑도 `스토리`, `캐릭터`, `세계관`, `서사`를 `story`로 보내도록 바꿨습니다.
- **보조 지표가 과도하게 노출되는 문제**: allowlist와 최소 근거 수, baseline 대비 변화량 기준을 적용해 실제로 두드러진 반응만 표시합니다.

## 🔀 통합 과정의 조정 (충돌 해결)

Groq 키 로테이션을 도입하면서 요약 로직과 같은 호출부가 겹쳐, 다음과 같이 조정했습니다.

| 파일 | 무엇을 조정했나 | 왜 |
|---|---|---|
| `reduce_api.py` | 구간 재호출의 `_run_feature_json(client=client)`를 `rotator=rotator`로 바꿨습니다. | Reduce가 `AsyncGroq`에서 `GroqKeyRotator`로 전환되었기 때문입니다. 이 덕분에 재호출도 키 로테이션 혜택을 받습니다. |
| `ai_service.py` | 기본 로컬 모델을 `gemma4:e4b`로, 키 입력을 `GROQ_API_KEYS`(없으면 `GROQ_API_KEY`)로 바꿨습니다. | gemma4 채택과 복수 키 로테이션을 함께 반영하기 위함입니다. |
| `chat_service.py` | 모듈 단위 단일 클라이언트(`_get_groq_client`)를 제거했습니다. | 챗봇이 키 로테이터로 요청마다 클라이언트를 만들게 되면서, 단일 재사용 클라이언트는 쓰이지 않는 코드가 되었기 때문입니다. |
| `run_map_pipeline.py` | Groq Map에 키 로테이션 루프와 `/no_think`(qwen3 추론 억제)를 함께 넣고, `_fetch_reviews`의 X-API-Key 인자와 버킷 처리 함수를 모두 유지했습니다. | 두 작업의 변경 사항을 모두 살리기 위함입니다. |
| `run_map_pipeline.py` / `ai_service.py` | Redis Map cache와 Reduce payload artifact 저장 경로를 추가했습니다. | 첫 로컬 Map/전체 재처리 결과를 보존해 Reduce 개선 시 Map 재실행 비용을 피하기 위함입니다. |
| `docker-compose.yml` | `14_migration_recommendation_targets.sql`을 초기화 스크립트로 마운트했습니다. | 새로 부팅할 때 추천 컬럼이 생성되도록 하기 위함입니다. |

## ⚙️ 통합 후 동작 변화 요약

| 항목 | 통합 전 | 통합 후 |
|---|---|---|
| 로컬 Map 모델 | qwen2.5:7b | gemma4:e4b |
| Map 라우팅 | 로컬 고정 | 자동(첫 작업은 로컬, 증분은 Groq) |
| Map 재사용 | 없음/경로별 상이 | Redis chunk cache + 첫/force Reduce payload artifact |
| Groq 키 | 단일 키 | 키 로테이션(복수 키, 429 시 전환) |
| 쓰기 엔드포인트 | 공개 | X-API-Key 인증 필요 |
| 추천 섹션 | 모든 게임 동일 문구 | 게임별 실제 데이터 |
| 게임 목록 | 소수 | 100개 타이틀 |
| 콘텐츠/스토리 | 한 항목에 혼재 | `content`와 `story` 분리 |
| 보조 지표 | 점수 중심 노출 | 근거 충분 + baseline 대비 두드러질 때만 방향 문구 노출 |

## ⚠️ 알려진 이슈 (Known Issues)

인증은 키가 설정되지 않으면 500 오류를 반환하도록 되어 있어 사실상 필수입니다. 그런데 로컬 `demo.py`는 아직 `run_map_pipeline.py`에 `--api-key`를 전달하지 않아 인증 엔드포인트와 연동되지 않습니다. 후속 작업으로 demo에 키를 전달하거나, 키가 없을 때는 개방하는 개발 모드를 추가할 필요가 있습니다.

## 검증 상태

py_compile, `app.main` import, `reduce_api`와 `key_rotator` import가 모두 통과했고, 충돌 4개 파일을 해결한 뒤 머지 커밋을 완료했습니다. 이후 `json_v6_story_aspect_split` Map 프롬프트 버전으로 스토리 aspect 분리와 보조 지표 노출 기준을 추가했습니다. 통합된 코드에 대한 E2E 검증(gemma4 풀-GPU, 인증 경유)은 별도 실행 로그로 추적합니다.
