# 캡스톤4분반_7조 — 게임 리뷰 AI 요약 서비스

Steam과 Metacritic에 흩어진 수백~수천 개의 게임 리뷰를, AI가 **근거 단위로 분해한 뒤 구매 판단에 쓰이는 지표로 재구성**해 한 화면에 정리하는 서비스입니다. 단순히 "AI가 요약한다"가 아니라, 리뷰 문장에서 `{리뷰 ID, 평가 항목, 긍·부정, 구체적 내용}` 형태의 근거를 뽑아내고, 이 근거를 유저/평론가·플레이타임 구간·평가 항목(aspect)·구매 시점이라는 축으로 나눠 저장·점수화하는 것이 핵심입니다.

LLM 파이프라인은 토큰이 많이 드는 **Map(근거 추출)** 과 품질이 중요한 **Reduce(최종 요약)** 로 나뉩니다. 대량 처리에서는 Map을 로컬 GPU에서 돌려 비용을 줄이고, 운영 증분 처리에서는 Map·Reduce를 모두 Groq로 처리해 클라우드 단독 실행도 가능합니다.

> 더 자세한 기술 설명(데이터 흐름, 점수 산출, DB·API, 배포)은 **[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)** 를 참고하세요.

## 1. 문제 정의

게임을 사기 전 사용자는 리뷰를 찾아보지만, 인기작 하나에도 리뷰가 수백·수천 개씩 달려 있고 길이도 내용도 제각각이라 "이 게임이 나에게 맞을까?"를 빠르게 판단하기 어렵습니다. 화면에 표시되는 평균 평점 하나로는 다음을 알 수 없습니다.

- 이 게임의 **구체적인** 장점과 단점이 무엇인지
- **오래 플레이해도** 재미가 유지되는지 (초반 호평이 후반까지 가는지)
- **평론가와 일반 유저**의 평가가 갈리는지
- **지금이** 살 만한 때인지 (할인 + 최근 여론)

이 서비스는 그 간극을, 리뷰를 근거 단위로 구조화해 메웁니다.

## 2. 핵심 차별점

| 차별점 | 설명 |
|---|---|
| **근거 기반 구조화** | 평균 평점이 아니라, 실제 리뷰 문장에서 `리뷰 ID·항목·긍부정·내용`을 추출해 장단점·추천 대상·항목별 점수를 구성합니다. 사용자에게 노출되는 모든 문장은 인용된 리뷰 근거를 갖습니다. |
| **유저·평론가 분리 저장** | 유저 리뷰와 평론가 리뷰를 별도 테이블·별도 LLM 호출로 요약하고, 점수도 서로 다른 근거로 독립 산출합니다. 한쪽 톤이 다른 쪽을 오염시키지 않습니다. |
| **플레이타임 구간 분석** | 리뷰 작성 시점 플레이 시간으로 초반·중반·후반을 나눠, "초반은 좋지만 후반에 반복적"처럼 시간에 따라 달라지는 평가를 포착합니다. |
| **항목별(aspect) 평가** | 그래픽·게임플레이·콘텐츠·스토리·최적화 등 9개 항목으로 분해하고, 콘텐츠 볼륨과 서사를 분리해 "할 거리"와 "스토리"가 한 점수에 섞이지 않게 합니다. |
| **점수의 출처 분리·검증** | 점수의 기준값은 공식 통계(Steam 추천률·Metacritic 평균)에 고정하고, LLM은 인용 근거가 검증된 보정값(delta)만 제안합니다. 코드가 최종 점수를 결정합니다. |
| **구매 시점 시그널** | 가격·할인과 최근 여론 변화를 종합해 "지금이 살 때인지"를 별도 지표로 제공하며, 사용자 트래픽이 외부 API 레이트리밋에 노출되지 않도록 스냅샷만 읽습니다. |
| **비용·성능 분리 실행** | 토큰이 많이 드는 Map은 로컬 GPU, 품질이 중요한 Reduce는 클라우드 LLM으로 처리합니다. |

## 3. 주요 기능

### 3-1. 리뷰 근거 기반 요약
Map 단계에서 리뷰 묶음마다 `{review_id, aspect, polarity, detail}` 형태의 근거(evidence)를 추출합니다. 복합 문장(예: "그래픽은 좋은데 최적화가 나쁘다")은 항목별로 쪼개고, 스포일러는 공개용 문장에서 가려집니다. Reduce 단계는 이 근거만을 토대로 한 줄 요약·장점·단점·키워드를 생성합니다.

### 3-2. 유저 / 평론가 분리 요약
- **유저 요약**(`user_summaries`): Steam 유저 리뷰 근거만으로 생성. 점수는 Steam 추천률을 기준점으로 산출합니다.
- **평론가 요약**(`critic_summaries`): Metacritic 평론가 리뷰만으로 생성. 점수는 Metacritic 평론가 평균을 사용합니다.
- 두 점수가 크게 갈리는 게임은 "숨은 호평작"·"구매 주의"로 구분할 수 있습니다(괴리 지표 API).

### 3-3. 플레이타임 구간별 분석
Steam 리뷰의 작성 시점 플레이 시간 분포에서 p33·p66 백분위로 초반·중반·후반 경계를 정하고, 각 구간을 별도로 요약합니다. 구간별 감성 점수·리뷰 수는 실제 추천 비율로 결정론적으로 산출합니다. 표본이 부족한 구간(기본 18건 미만)은 데이터 부족으로 정직하게 비웁니다.

### 3-4. 항목별(aspect) 평가 + 레이더 차트
`graphics, controls, optimization, content, story, price_value, sound, gameplay, difficulty` 9개 항목으로 분해합니다. 프론트는 공통 5축(`content·gameplay·graphics·controls·optimization`)을 레이더로 고정하고, 나머지 항목은 근거가 충분하고 리뷰에서 두드러질 때만 "눈에 띄는 반응"으로 노출합니다. 레이더 면적은 0~10 항목 점수의 절대 크기를 따르고, 색·라벨은 종합 평가와 같은 Steam식 9밴드(압도적으로 긍정적 … 복합적 … 부정적)를 0~10 점수에 환산해 표시합니다. 별도의 대표 강점/약점 캡션은 전체 리뷰의 언급량·긍정률(`category_frequency`) 기반 라벨을 사용해, **평가 강도**와 **자주 회자된 이슈**를 분리합니다.

### 3-5. 추천 / 주의 대상
유저 요약 단계에서 LLM이 리뷰 근거를 바탕으로 "이런 사람에게 추천 / 주의가 필요한 사람"과 그 사유를 생성합니다. 과거 모든 게임에 동일했던 카테고리 고정 문구를 실데이터 기반으로 교체한 것입니다.

### 3-6. 구매 시점 / 가격 시그널
할인율과 최근 월별 여론 변화(긍정 비율 회복/부정 급증)를 종합해 구매 타이밍을 표시합니다. 가격·여론 스냅샷은 일일 잡이 Redis에 미리 채워 두고, 조회 API는 그 스냅샷만 읽으므로 사용자 요청이 Steam 레이트리밋에 노출되지 않습니다. 스냅샷이 오래되면(28시간 초과) 할인을 단정하지 않고 "스토어에서 확인 권장"으로 낮춥니다.

### 3-7. AI 추천 챗봇 *(보조 기능)*
화면 우측 하단의 챗봇은 "좋아하는/싫어하는 게임"을 입력받아 추천을 제공합니다. **핵심 아키텍처의 중심축이 아니라, 위에서 만든 요약 데이터를 활용하는 보조 사용자 기능**입니다.

- 추천 근거는 **DB에 수록된 게임 데이터로 제한**됩니다. system prompt에 게임별 태그·한줄평·장점·단점·키워드·추천 대상·Steam 추천률을 카탈로그로 주입하고, 목록에 없는 게임이나 게임 추천 외 주제는 거절하도록 강제합니다(외부 지식 환각 차단).
- 게임 카탈로그는 Redis에 5분 캐시, 요청은 IP 기준 분당 10회로 제한, Groq 키 로테이션으로 한도 초과에 대응합니다.

## 6. 실행 방법

### 6-1. 로컬에서 빠르게 띄우기

```bash
cp .env.example .env            # GROQ_API_KEY(또는 GROQ_API_KEYS), API_SECRET_KEY 등 입력
docker compose up -d --build    # 백엔드 스택: postgres · redis · backend · scheduler · frontend · adminer

# (선택) 로컬 GPU에서 Map을 돌릴 경우에만 — ollama는 메인 compose에 없음
docker compose -f docker-compose.map.yml up -d
docker exec capstone_ollama_map ollama pull gemma4:e4b
```

- 프론트엔드: `http://localhost`
- API 문서(Swagger): `http://localhost:8000/docs`
- DB 관리(adminer): `http://localhost:8888`

쓰기 엔드포인트(리뷰 적재·요약 트리거)는 `X-API-Key`(= `API_SECRET_KEY`) 인증을 요구합니다.

### 6-2. 데이터 수집 → 적재 → 요약

```bash
# 크롤링 (결과는 crawling/output/ 에 저장)
python crawling/steam/steam_crawler.py
python crawling/metacritic/metacritic_crawler.py

# 백엔드로 적재 (X-API-Key 필요)
python crawling/send_to_api.py steam
python crawling/send_to_api.py metacritic

# 특정 게임 요약 트리거 (force=true면 커서 무시하고 전체 재처리)
curl -X POST "http://localhost:8000/api/v1/games/{game_id}/summarize?force=true" \
     -H "X-API-Key: <API_SECRET_KEY>"
```

## 7. 운영 / 배포 방식 요약

이 서비스의 Reduce는 항상 Groq에서 실행되지만, **Map은 두 방식 중에서 선택**합니다(`MAP_BACKEND`, compose 기본값 `groq`).

| 모드 | Map 실행 위치 | 언제 |
|---|---|---|
| **모드 A — 클라우드 단독** (`MAP_BACKEND=groq`) | 클라우드 백엔드 안에서 Groq API 호출 | 일상 증분 요약. GPU 불필요. |
| **모드 B — 로컬 GPU Map** (`run_map_pipeline.py`) | 로컬 GPU의 Ollama | 첫 요약·전체 재처리처럼 리뷰가 많아 토큰이 폭증하는 작업. |

- **모드 A (스케줄러 자동)**: `scheduler` 컨테이너가 매일 17:05 UTC에 **① 가격·여론 스냅샷 갱신 → ② Steam 증분 리뷰 크롤 → ③ AI 증분 요약**을 한 타임라인으로 직렬 실행합니다. 세 잡은 각각 독립 try로 감싸 한 잡 실패가 다음 잡을 막지 않습니다. 증분 정확성은 ingestion의 `(platform, game, 리뷰키)` upsert가 보장하므로(중복 흡수, 신규만 새 ID), 매일 같은 최신 윈도우를 재전송해도 안전합니다.
- **모드 B (로컬 GPU + Cloudflare Tunnel)**: 로컬 머신이 클라우드 백엔드의 공개 URL로 접속해 `① /reviews-for-map`으로 리뷰를 받고 → 로컬 Ollama로 Map → `② /reduce`로 근거를 보내 클라우드가 Reduce·저장합니다. 첫/`--force` 실행은 Reduce 입력 전체를 artifact로 보존해, Reduce 로직만 바꾼 뒤 `--from-payload`로 Map 없이 재실행할 수 있습니다.

```bash
# 모드 B 실행 예
python run_map_pipeline.py \
  --cloud-url https://xxx.trycloudflare.com \
  --api-key <API_SECRET_KEY> \
  --map-route auto --all
```

`--map-route auto`는 리뷰가 많은 배치는 로컬 Map, 작은 증분은 Groq Map으로 자동 분배하므로 로컬 GPU가 꺼져 있어도 클라우드만으로 증분 갱신이 됩니다.

## 8. 현재 구현 상태와 알려진 주의 사항

**구현 완료 (코드 기준)**

- Steam/Metacritic 크롤 → 적재 → Map-Reduce 요약 → 프론트 서빙 전 경로
- 유저/평론가/플레이타임/항목별 분리 요약 및 독립 점수 산출
- 추천·주의 대상, 구매 시점 시그널, 유저·평론가 괴리 지표
- AI 추천 챗봇(DB 게임 한정 근거), 대표 리뷰 번역, 게임 비교 화면
- 일일 스케줄러(가격·여론·증분 크롤·요약 직렬화)

**알려진 주의 사항**

- **DB 마이그레이션**: `docker-compose.yml`은 `database/*.sql`을 첫 부팅(빈 볼륨) 시에만 자동 적용합니다. 또한 compose에는 14번까지만 마운트돼 있어, `15_migration_steam_rating.sql`은 기존 볼륨에 **수동 `ALTER`로 적용**해야 합니다(`database/` 참고).
- **종합 등급 라벨의 출처**: 화면의 9밴드 등급(압도적으로 긍정적 등)은 **계산된 감성 점수**에서 Steam 공식 컷으로 도출합니다. 점수의 기준값(baseline)은 Steam 공식 `query_summary` 추천률에 고정되며, 증분(모드 A)·전체(모드 B) 실행 모두 이 공식 집계로 보강됩니다(Steam 호출 실패 시 그 실행만 수집 표본으로 폴백). 따라서 등급은 "공식 등급 그대로 표기"가 아니라 "공식 통계를 기준점으로 한 LLM 산출 점수의 밴드 매핑"입니다.
- **항목별(aspect) 표시 기준 분리**: 레이더 축은 항목 점수(0~10)를 0~100으로 환산해 종합 평가와 같은 9밴드 라벨로 표시합니다. 반면 대표 강점/약점 라벨은 전체 리뷰에서 집계한 항목별 **언급량 + 긍정률**(`category_frequency`)로 정합니다 — 강점은 자주 회자되며(언급 ≥30) 긍정률이 높은(≥0.92) 정의적 특징만, 약점은 충분히 언급되며(≥12) 긍정률이 낮은(≤0.78) 진짜 불만만 잡습니다. 이렇게 점수 기반 평가 강도와 리뷰 담론 기반 대표 이슈를 분리해, "전반적으로 높은 점수지만 특별히 자주 회자되지는 않는 항목"과 "점수는 보통이어도 반복적으로 지적되는 항목"을 구분합니다.
- **증분 품질 가드**: 저품질·소량 신규 배치가 기존 양질 요약을 덮지 않도록 ① 최소 신규 건수 미만이면 스킵, ② 직전 근거를 합쳐 최근 편향 완화, ③ 빈약한 결과면 기존 요약 보존하는 가드가 있습니다(자세히는 ARCHITECTURE §5-7).
- **가격 정보**: 가격은 DB가 아니라 Redis에만 보관(TTL 약 30시간)되며 `price_refresher` 잡이 채웁니다. Redis를 비우거나 잡을 돌리지 않으면 가격이 "갱신 대기 중"으로 표시됩니다.
- **챗봇 범위**: 챗봇은 DB에 수록된 게임만 다루도록 제한됩니다. 일반 지식이나 미수록 게임 질문은 의도적으로 거절합니다.
- **Metacritic 자동 크롤 제외**: 평론가 리뷰는 거의 고정이고 Playwright 비용이 커서 일일 스케줄러에서 제외하며 필요 시 수동 수집합니다.
- **CORS**: 현재 모든 출처를 허용(`allow_origins=["*"]`)하므로 실제 공개 배포 시에는 도메인을 제한해야 합니다.
