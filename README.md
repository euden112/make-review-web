# 캡스톤4분반_7조

## 브랜치 구조

```
main
```

중간 발표 시점까지는 `main` 단일 브랜치로 운영했습니다.

팀원별로 담당 파트(크롤링 / 백엔드 / AI 파이프라인)가 명확히 분리되어 있어
작업 간 충돌 가능성이 낮았고, 별도 브랜치 없이 각자 맡은 파트를 `main`에 직접 커밋하는 방식으로 진행했습니다.

### 향후 브랜치 전략 (기말 발표 대비)

기말 발표를 위한 기능 확장부터는 `feature/*` 브랜치를 분기하여 `main`에 Pull Request로 병합하는 방식을 적용합니다.
중간 발표까지 완성된 데모가 `main`에 있으므로, 신규 작업이 기존 동작에 영향을 주지 않도록 보호하기 위함입니다.

```
main
├── feature/multilang-pipeline    # 다국어(ko/zh) 파이프라인 구현
├── feature/frontend-integration  # 프론트엔드 백엔드 연동
└── fix/xxx                       # 버그 수정
```

- 작업 단위로 `feature/*` 또는 `fix/*` 브랜치를 생성
- 개발 완료 후 Pull Request를 통해 팀원 리뷰 후 `main` 병합
